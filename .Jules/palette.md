## 2026-04-06 - Explicit Disabled State Communication for Multi-Conditional Actions
**Learning:** In the DJ app, buttons like 'Transition' can be disabled for multiple reasons (e.g., currently generating a transition vs. missing tracks on decks). Relying solely on a visually disabled state without context leaves users confused about what action is required to enable the button.
**Action:** When a button's disabled state depends on complex application logic or multiple conditions, always provide a dynamic `title` tooltip that explicitly states *why* it is disabled and *what* the user needs to do (e.g., 'Load tracks on both decks to transition' instead of a generic disabled state), and pair it with explicit Tailwind `disabled:opacity-50 disabled:cursor-not-allowed` classes to clearly signal the state visually.
